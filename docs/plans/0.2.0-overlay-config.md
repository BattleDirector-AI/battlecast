# 0.2.0 release plan — Broadcaster config / UX

**Theme:** deliver the overlay-configuration epic (#19) — a broadcaster can arrange
the `/all` overlay, configure a rotating logo carousel, and save/load it for OBS,
without editing code or CSS. This attacks the single most-cited friction of the
native rF2/LMU overlays (no WYSIWYG editor) and adds logo rotation (native in rF2,
a gap in LMU). See `docs/research/native-overlays.md`.

## The gating decision (#32)

battlecast is a stateless static app today. Two separable problems:

- **Config** (widget geometry, visibility, theme, rotation timing, producer URL) is a
  few KB of JSON — trivial to persist and load.
- **Logo images** are binary assets a JSON blob can't hold — this is what decides the
  architecture.

Candidate approaches (evaluated in #32): (1) fully static with downloadable
`config.json` + user-placed image folders; (2) client-only localStorage/IndexedDB;
(3) a light companion server serving the static app plus a small config/asset API.

**Working recommendation (to confirm in the spike):** approach (3), a light companion
server. It matches the model rF2 and every third-party tool converge on (transparent
overlay page + separate control panel + local server), serves config and images
uniformly, and supports the drag-drop upload UX that differentiates us. The OBS render
page loads a profile by id (`/all?profile=<name>`) and fetches its config + asset URLs
from the server. If the spike finds the server cost unjustified, fall back to approach
(1) — still viable, at the cost of in-UI image upload. **No work on #16/#33/#34 starts
until #32 records the decision on #19.**

## Sequencing

```
#32 spike (decide config + asset model)   ── gate ──┐
                                                     ▼
        ┌──────────────────┬───────────────────┬──────────────────┐
        ▼                  ▼                   ▼                  
  #16 layout          #33 logo            #34 config UI
  (render side:       rotation widget     (edit side: drag/resize/
  read config,        (consumes image      toggle, logo mgmt,
  place widgets)      set + timing)        save/load profiles)
        └──────────────────┴───────────────────┘
                                                     ▼
                              integration + fixture-based tests on /all
                                                     ▼
                                    0.2.0 release (next → prerelease → main)
```

- **#32 first**, alone — it defines the config schema and asset story every other
  item depends on. Land the config-shape definition (a documented config object +
  loader) as its concrete output so #16/#33/#34 build against a fixed contract.
- After the gate, **#16, #33, #34 proceed in parallel**; #34 (edit side) and #16
  (render side) meet at the shared config contract, so define that contract in #32
  and both sides target it.
- **Integration pass** wires the config UI → saved profile → `/all` render and adds
  end-to-end behavioral tests before release.

## Testing (per CONTRIBUTING — fixture-based behavioral)

- **Config→render**: given a config fixture (widget geometry + visibility), assert the
  rendered `/all` places/sizes/hides widgets accordingly.
- **Logo rotation**: given an image set + durations, assert which image is shown across
  a sequence, and the empty-state render.
- **Profile round-trip**: save a profile, reload it, assert identical render.
- Keep the spec-v1 compliance harness green (no contract change in 0.2.0).

## Exit criteria (release gate)

- [ ] #32 decision recorded on #19; config contract documented.
- [ ] #16, #33, #34 complete with passing behavioral tests.
- [ ] A broadcaster can arrange widgets + configure logo rotation, save a profile, and
      load it in an OBS Browser Source with no code/CSS edits.
- [ ] CI green (app lint/test/build + compliance) on `main` after promotion.
- [ ] Version bumped to `0.2.0`, promoted `next → prerelease → main`, tagged `v0.2.0`
      with release notes.

## Risks

- **Introducing state** to a static app (if approach 3) — keep the server minimal and
  optional for the pure-render path; the render page must still work from a static
  config for users who don't need the UI.
- **Config contract churn** — pin it in #32 so parallel work doesn't diverge.
- **OBS Browser Source context** — the render page must get its config without relying
  on the editor's browser storage; the spike must validate the profile-load path end to
  end in an actual Browser Source.
