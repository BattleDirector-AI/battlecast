# 0.5.0 release plan — Spec v1.x additive fields + session/flag status widget

**Theme:** open the critical path for the remaining core-parity widgets by landing the
first slice of the additive spec v1.x field batch (#20), then prove the pipeline
end-to-end by shipping the widget with the smallest field surface and highest broadcast
value: the **session / flag / FCY / SC status** overlay (#25). See
`docs/research/native-overlays.md` and epic #31.

**Milestone:** [0.5.0](https://github.com/BattleDirector-AI/battlecast/milestone/5)
· **Epic:** #31 (widget parity) · **Spec slice:** #90 (slice 1 of #20)
**Built on:** 0.4.0 (results #23, grid #24, class-aware tower #28, re-cut reveal #64/#73,
auto-cycling mock #66).

## Why this, why now

Every widget shipped through 0.4.0 rendered from data spec v1 **already carried**. Every
remaining core-parity widget (#25/#26/#27/#61/#62/#63) needs **new** fields, so #20 is
the gate for all of them. Rather than land the whole #20 batch speculatively, we take the
epic's own guidance — *"additive fields, prioritize per widget demand"* — and cut the
thinnest useful slice: the **session-level fields**, paired with their consumer (#25).
This validates the additive-field mechanics (schema + SPEC + fixtures + mock + compliance,
per CONTRIBUTING) on a small surface before committing to the larger telemetry (#26) and
track-position (#29/#63) slices.

## The additive contract (#20, slice 1)

A new **optional, top-level `session` object**. Consistent with how `gap_to_leader` and the
`notable` flags were added — `additionalProperties` is already `true`, consumers tolerate
absent/unknown fields (SPEC "Versioning policy"), and **`schemaVersion` stays `"1"`**
(additive, non-breaking). *Dumb overlay, smart producer* holds: the producer owns every
judgment below; the widget only renders.

```jsonc
// top-level, alongside vehicles / subject / relationship — all fields OPTIONAL
"session": {
  "flag": "green",            // string: green|yellow|red|checkered|white|none.
                              //   Free-form; consumers tolerate unknown values.
  "full_course_yellow": false,// boolean. FCY active. Distinct from a local yellow `flag`.
  "safety_car": false,        // boolean. Safety Car deployed.

  // --- session progress: a session is TIMED or LAP-LIMITED; the widget auto-selects ---
  "time_remaining": 1732.4,   // number|null — seconds left in a timed session.
  "session_length": 3600,     // number|null — total timed length, for a progress render.
  "laps_remaining": null,     // integer|null — laps left in a lap-limited session.
  "total_laps": null,         // integer|null — scheduled lap count (the Y in "lap X of Y").
  "current_lap": null,        // integer|null — producer-owned lap the leader is on (the X
                              //   in "lap X of Y"); rendered verbatim, never derived.
  "basis": null               // string|null — optional explicit "time" | "laps"
                              //   override. Absent/null => consumer infers (see below).
}
```

Notes:
- **`flag` vs `full_course_yellow`/`safety_car`**: `flag` is the trackwide flag color;
  FCY and SC are separate boolean states that can coexist with a yellow flag. Keeping them
  discrete lets the widget show the right combination without overloading one enum.
- **Timed vs lap-limited — accommodated automatically.** Rather than force the producer to
  declare a mode, the consumer **infers** it from which progress fields are populated:
  1. If `basis` is set (`"time"` / `"laps"`), honor it.
  2. Else if `time_remaining` is non-null → render the **countdown clock** (with
     `session_length` as the progress denominator when present).
  3. Else if `laps_remaining` / `total_laps` / `current_lap` is non-null → render the
     **lap counter** ("LAP X OF Y" from the producer-owned `current_lap` + `total_laps`,
     rendered verbatim; else the raw laps-remaining when no `current_lap` is supplied).
  4. Else → hide the progress element; flag/FCY/SC still render.

  A timed session simply leaves the lap fields null and vice-versa, so the common case
  needs no `basis` and no producer coordination — it just works from whichever fields
  carry data.
- **Endurance "time-certain + N laps"** (race runs to a clock, then completes the lead
  lap): near the end **both** may be non-null. This is the one ambiguous case, so the
  producer resolves it — set `basis:"laps"` (or null `time_remaining`) once the leader
  starts the final timed lap, and the widget flips clock → lap counter automatically.
- Producer owns FCY/SC/flag/clock/laps; the consumer MUST NOT derive them from
  `vehicles[]`. In particular the **lap number is producer-owned** (`current_lap`): the
  counting convention (does the lap in progress count? how is the final lap numbered?) is
  the producer's judgment, so the widget renders `current_lap` verbatim and never computes
  it from `total_laps` − `laps_remaining`.

Deliverables for the slice (all in one PR set, per CONTRIBUTING):
`spec/v1/schema.json` (add `session` `$def`), `spec/v1/SPEC.md` (field walkthrough +
timed/lap note), fixtures under the compliance harness covering present / absent / partial
`session`, and the **mock producer** emitting `session` across its phase cycle
(green race → FCY → SC → checkered at results). Compliance harness stays green.

## The consumer widget (#25)

A session/flag status overlay on its own route (`app/src/routes/session/` +
`SessionStatusPage.svelte`), mirroring the existing widget conventions:

- Renders the current flag state, and FCY / Safety-Car indicators when active, plus the
  session clock (`time_remaining` countdown, or `laps_remaining`) when present.
- Explicit **idle state** — "Waiting for session…" with no snapshot — so it never blanks
  (same discipline as the results/grid slides).
- **Reduced-motion gating** and the on-brand transition vocabulary (skewed bar-wipe reveal;
  see the lower-third standard) for state changes — a flag change should reveal, not pop.
- Tolerates a total absence of the `session` object (older producers) by staying in idle
  rather than erroring.

## Testing (per CONTRIBUTING — fixture-based behavioral)

- **Field presence matrix**: `session` present / absent / partial (e.g. `flag` only,
  clock only) each render a sensible frame; absent → idle, never a crash.
- **Flag/FCY/SC combinations**: green, local yellow, FCY-over-yellow, SC, red, checkered
  each render their expected indicator set.
- **Progress auto-selection**: timed-only (`time_remaining`) → clock; lap-only
  (`laps_remaining`/`total_laps`) → lap counter ("lap X of Y" when `total_laps` present);
  both present + `basis:"laps"` → lap counter (endurance flip); neither → progress hidden,
  rest of widget intact. Assert the inference precedence and the explicit `basis` override.
- **Reduced-motion**: assert the real-motion reveal path and the reduced-motion static
  path (stub `matchMedia`; the test env defaults reduced-motion to true).
- **Compliance harness stays green** — the `session` additions are additive, `schemaVersion`
  unchanged; assert a spec-v1 payload with **no** `session` still validates.

## Exit criteria (release gate)

- [ ] `session` object defined in `schema.json` + documented in `SPEC.md`; compliance
      harness green with and without it; `schemaVersion` still `"1"`.
- [ ] Mock producer emits `session` across all phases (incl. an FCY and an SC window, a
      timed race phase and a lap-limited phase so both progress modes are exercised).
- [ ] #25 widget renders flag/FCY/SC/clock, has an idle state, is motion-gated, and
      tolerates an absent `session`.
- [ ] Behavioral tests above passing; app lint/test/build green.
- [ ] CI green on `main` after promotion.
- [ ] Version bumped to `0.5.0`, promoted `next → prerelease → main`, tagged `v0.5.0`
      with release notes; epic #31 checklist updated (#25 done, #20 slice 1 done).

## Risks

- **Batch creep on #20** — the temptation to land all of #20 at once. Mitigation: this
  plan scopes *only* the `session` slice; other slices are explicitly deferred to their
  consumers. Keep the PR to the session fields.
- **Flag-state modeling** — real broadcasts combine flag + FCY + SC in ways one enum can't
  capture. Mitigation: discrete boolean FCY/SC alongside a free-string `flag`; producer
  owns the combination, widget renders whatever it's given.
- **Un-consumed fields** — adding spec fields no widget reads. Mitigation: the widget ships
  in the same release; the mock is the reference producer that keeps the fields exercised.
