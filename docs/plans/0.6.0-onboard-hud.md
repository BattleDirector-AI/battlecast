# 0.6.0 release plan — Spec v1.x live-input telemetry + on-board HUD widget

**Theme:** land the **second slice** of the additive spec v1.x field batch (#20) — the
per-subject **live-input telemetry** (throttle / brake / speed / gear) — paired with its
consumer, the over-camera **on-board HUD** (#26). This is the exact same additive-field
mechanics proven in 0.5.0 (the `session` slice #90 + Race Control widget #25), applied to
the next widget on the core-parity critical path (epic #31). See
`docs/plans/0.5.0-spec-fields-session-status.md` for the template this plan follows.

**Milestone:** [0.6.0](https://github.com/BattleDirector-AI/battlecast/milestone/6)
· **Epic:** #31 (widget parity) · **Spec slice:** #102 (slice 2 of #20)
**Built on:** 0.5.0 (`session` object #90, Race Control widget #25, session clock in the
tower header #94).

## Why this, why now

0.5.0 validated the additive-field pipeline (schema + SPEC + fixtures + mock + compliance)
on the session slice. This release takes the next slice named in that plan's own follow-on
table — *"Live input telemetry → On-board HUD #26"* — and ships it the same way: the field
lands **next to the widget that reads it**, so no field ships un-consumed and the compliance
fixtures always have a real reader. The on-board HUD is the first widget that reads a
**live, every-tick** input channel rather than cut-driven identity data, so it exercises a
new axis of the *dumb overlay, smart producer* contract: high-frequency producer values the
overlay renders verbatim each frame.

## The additive contract (#20, slice 2)

A new **optional `telemetry` sub-object on `subject`**. Consistent with how `session` and
the vehicle-level optional fields were added — `additionalProperties` is already `true` at
every level, consumers tolerate absent/unknown fields (SPEC "Versioning policy"), and
**`schemaVersion` stays `"1"`** (additive, non-breaking). *Dumb overlay, smart producer*
holds: the producer owns every value; the widget only renders.

```jsonc
// on `subject`, alongside slot_id / driver_name — the whole object and every field OPTIONAL
"subject": {
  "slot_id": "car-1",
  "driver_name": "Verstappen",
  "telemetry": {
    "throttle": 0.82,   // number 0–1|null — throttle application (0 = closed, 1 = full).
    "brake": 0.0,       // number 0–1|null — brake application (0 = off, 1 = full).
    "speed": 247,       // number|null — vehicle speed; PRODUCER-DEFINED UNIT (see SPEC).
    "gear": 6           // integer|null — selected gear; 0 = neutral, -1 = reverse (convention).
  }
}
```

### Why a `subject.telemetry` sub-object (not flat fields on `subject`)

**Decision: group the four inputs under `subject.telemetry`.** Two reasons:

1. **Churn isolation.** `subject.slot_id` / `driver_name` are stable identity that changes
   only on a camera cut; throttle/brake/speed/gear change *every tick*. Grouping the
   high-frequency channel in its own labelled object keeps the two cadences visually and
   structurally distinct in the payload, and signals to a producer author that this block
   is the live-input feed (populate it every frame) versus the identity fields (set on cut).
2. **Namespacing for growth.** The telemetry channel is the natural home for future
   live-input fields (RPM, DRS, steering, clutch) without crowding `subject`'s top level.
   `subject.telemetry` is `additionalProperties: true` so those land additively too.

The whole sub-object is optional: a producer that has no live telemetry (or a phase where
the car is parked) simply omits it, and the HUD idles. Every field within is independently
optional so a partial feed (e.g. throttle + speed but no gear) still renders what it has.

Deliverables for the slice (all in one PR set, per CONTRIBUTING):
`spec/v1/schema.json` (add a `telemetry` `$def`, referenced from `subject`),
`spec/v1/SPEC.md` (field walkthrough + live-input note), fixtures under the compliance
harness covering present / partial `telemetry` (and the existing no-`telemetry` fixtures as
the backward-compat case), and the **mock producer** emitting `subject.telemetry` for the
on-camera subject each tick in the running phases. Compliance harness stays green; a payload
with **no** `telemetry` still validates (asserted directly in the harness self-test).

## The consumer widget (#26)

The **on-board HUD** (`app/src/routes/onboard/OnBoardHud.svelte`, route **`/onboard`**,
composed into `/all`) — an over-camera overlay for the on-camera subject's live inputs:

- **Throttle / brake bars** — two fill bars driven by `telemetry.throttle` /
  `telemetry.brake` (each clamped to [0, 1]); throttle green, brake red, the two colours
  broadcast convention. Update every tick.
- **Speed readout** — the `telemetry.speed` numeral (rounded), under a neutral `SPEED`
  label (no unit claim; producer-defined unit).
- **Gear indicator** — the `telemetry.gear` numeral, with `N` for `0` (neutral) and `R`
  for `-1` (reverse) by convention.
- **Live, not cut-driven.** Unlike the driver/qualifying lower-thirds (which fire on a
  camera cut and dwell), the HUD re-renders the subject's inputs on *every* snapshot — it
  reads `snapshot.subject.telemetry` directly, no dwell/trigger machinery.
- **Idle tolerance.** Renders nothing when there is no telemetry to show (absent sub-object,
  or a phase where the car is parked), and tolerates partial/garbage field types without
  throwing — same *dumb overlay* discipline as Race Control.

The route mirrors the racecontrol/tower structure: `OnBoardHudPage.svelte` (SSE wiring) +
`OnBoardHud.svelte` (component, with a `resolveTelemetry` module helper) + `sseClient.js` +
`OnBoardHud.test.js`. Composed into `/all` via `AllView.svelte` + a new `onboard` entry in
`DEFAULT_CONFIG`, covered by `AllLayout.test.js`.

## Testing (per CONTRIBUTING — fixture-based behavioral)

- **Field presence matrix**: `telemetry` present (all four) / absent / partial (e.g.
  throttle + speed only) each render a sensible frame; absent → idle (renders nothing),
  never a crash.
- **Bars**: throttle/brake fill widths track their [0, 1] values; values are clamped
  (out-of-range and garbage tolerated).
- **Gear**: `0` → `N`, `-1` → `R`, positive integers verbatim; absent gear hidden.
- **Speed**: rounded numeral rendered; absent speed hidden.
- **/all composition**: the HUD composes into `/all` from `snapshot.subject.telemetry`
  without disturbing the other widgets; hidden via config removes it from the DOM.
- **Compliance harness stays green** — the `telemetry` additions are additive,
  `schemaVersion` unchanged; assert a spec-v1 payload with **no** `telemetry` still
  validates.

## Risks

- **Batch creep on #20** — the temptation to land more of #20 (RPM, DRS, unit labels) at
  once. Mitigation: this plan scopes *only* the four throttle/brake/speed/gear fields; the
  rest is explicitly deferred. Keep the PR to those fields.
- **Unit ambiguity on `speed`** — no unit is declared. Mitigation: document `speed` as a
  producer-defined-unit number and render a neutral `SPEED` label; a unit field is a future
  additive slice if needed.
- **Un-consumed fields** — adding spec fields no widget reads. Mitigation: the HUD ships in
  the same release; the mock is the reference producer that keeps the fields exercised each
  tick.
