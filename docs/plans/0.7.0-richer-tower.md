# 0.7.0 release plan — Spec v1.x richer per-vehicle metrics + LMU-style standings tower

**Theme:** land the **third slice** of the additive spec v1.x field batch (#20) — the
**richer per-vehicle timing/strategy metrics** (interval-to-next, pit count, in-pit,
tire, fuel/energy) — paired with its consumer, the **LMU-style richer standings tower**
(a follow-on to #28, which shipped gap-to-leader in 0.4.0). This is the exact same
additive-field mechanics proven in 0.5.0 (the `session` slice #90 + Race Control widget
#25) and 0.6.0 (the `subject.telemetry` slice #102 + on-board HUD #26), applied to the
next fields on the core-parity critical path (epic #31). See
`docs/plans/0.6.0-onboard-hud.md` for the template this plan follows.

**Milestone:** [0.7.0](https://github.com/BattleDirector-AI/battlecast/milestone/7)
· **Epic:** #31 (widget parity) · **Spec slice:** #110 (slice 3 of #20)
· **Widget:** #111 (richer tower, follow-on to #28).
**Built on:** 0.6.0 (`subject.telemetry` #102, on-board HUD #26), 0.5.0 (`session` #90,
Race Control #25), and 0.4.0 (tower gap-to-leader #28/#75).

## Why this, why now

0.5.0 and 0.6.0 validated the additive-field pipeline (schema + SPEC + fixtures + mock +
compliance) twice over — once on a session sub-object, once on a `subject` sub-object.
This release takes the slice named in both plans' own deferral tables — *"richer
per-vehicle metrics (interval-to-next, pit count, tire, fuel/energy) → LMU-style richer
tower follow-on to #28"* — and ships it the same way: the fields land **next to the
widget that reads them**, so no field ships un-consumed and the compliance fixtures always
have a real reader. The standings tower already reads a per-vehicle every-tick channel
(`gap_to_leader`, 0.4.0); this slice broadens that same row into the LMU/rF2 timing-tower
feature set — a second timing interval plus the strategy metrics an endurance broadcast
leans on.

## The additive contract (slice 3 of #20)

Six new **optional fields on `vehicle`**. Consistent with how `gap_to_leader` /
`car_number` / `make` / `model` were added — `additionalProperties` is already `true` at
every level, consumers tolerate absent/unknown fields (SPEC "Versioning policy"), and
**`schemaVersion` stays `"1"`** (additive, non-breaking). *Dumb overlay, smart producer*
holds: the producer owns every value; the tower only renders.

```jsonc
// on each `vehicle`, alongside gap_to_leader / car_number — every field OPTIONAL
{
  "slot_id": "car-1",
  "gap_to_leader": 0.400,   // existing (0.4.0): interval to the OVERALL leader
  "interval_ahead": 0.400,  // number|null — gap in seconds to the car immediately AHEAD.
  "pit_stops": 1,           // integer|null — count of COMPLETED pit stops this session.
  "in_pit": false,          // boolean — true while in the pit lane/box right now.
  "tire_compound": "M",     // string|null — compound label, rendered verbatim (S/M/H/W…).
  "tire_wear": 0.35,        // number 0–1|null — normalized wear, 0 = fresh, 1 = worn out.
  "fuel": 0.62              // number 0–1|null — normalized fuel/energy remaining, 1 = full.
}
```

### Why flat fields on `vehicle` (not a grouped sub-object)

**Decision: keep the six metrics flat on `vehicle`.** This is the deliberate *opposite* of
the `subject.telemetry` grouping decision, and for a principled reason — the two objects
have different shapes:

1. **Consistency with the existing `vehicle` shape.** Every field on `vehicle` today is
   flat and already spans every churn cadence: `gap_to_leader` moves every tick, `best_lap`
   once per lap, `car_number` / `make` / `model` once per session. `vehicle`'s only
   sub-object is `notable`, which groups *homogeneous boolean flags*, not a churn split.
   Adding a metrics sub-object would introduce an inconsistency the payload does not
   currently have; adding flat fields is more of exactly what is already there.
2. **The telemetry grouping's rationale does not transfer.** `subject.telemetry` was
   grouped because `subject` had a clean **stable-identity vs high-churn** split — two
   identity fields (`slot_id` / `driver_name`) that change on a camera cut versus four
   inputs that change every frame. `vehicle` has no such clean split: `interval_ahead`,
   `tire_wear`, and `fuel` drift every tick, while `pit_stops`, `in_pit`, and
   `tire_compound` change only at a pit stop. A single grouping would not isolate churn
   cleanly, so it buys none of the churn-isolation benefit that justified `telemetry`.
3. **`interval_ahead` is a timing-column sibling.** It belongs flat right beside
   `gap_to_leader` — the two are the tower's "Gap" and "Int" columns — not buried in a
   strategy object.
4. **Lightweight namespacing without ceremony.** `tire_compound` / `tire_wear` share a
   `tire_` name prefix — enough of a namespace to read the tire cluster at a glance,
   without the nesting a sub-object would impose. Future strategy fields land additively
   the same way.

Every field is independently optional: a producer with no strategy data simply omits them
and the tower renders its timing rows exactly as today; a producer that has, say, tire but
not fuel populates what it has and the tower shows what it is handed.

**`interval_ahead` semantics (producer-owned, like `gap_to_leader`).** The gap in seconds
to the car immediately ahead **in overall running order**. The producer owns the metric
(an on-track time gap in a race, a lap-time delta in a qualifying-style session); consumers
render it verbatim and MUST NOT derive it by scanning `vehicles[]`. Null/absent for the
leader (no one ahead) or until determined. Unlike the grouped tower's gap-to-*class*-leader
(exact arithmetic the spec permits on `gap_to_leader`), a class-relative interval is **not**
re-derivable, so a grouped tower renders `interval_ahead` verbatim (gap to the car ahead
overall) and makes no class-relative claim.

**`tire_wear` / `fuel` are normalized `[0, 1]`.** Both are producer-computed fractions —
`tire_wear` 0 = fresh → 1 = fully worn; `fuel` 1 = full → 0 = empty — so the tower renders
each as a bar (clamped defensively) with no unit claim, the same discipline the throttle /
brake bars take. `fuel` is deliberately one field covering **fuel or hybrid energy**
(producer-defined which); the overlay shows a neutral resource bar and makes no ICE-vs-EV
claim.

Deliverables for the slice (all in one PR set, per CONTRIBUTING):
`spec/v1/schema.json` (six properties added under the `vehicle` `$def`),
`spec/v1/SPEC.md` (field walkthrough in the vehicle optional-fields list),
fixtures under the compliance harness covering present / partial metrics (and the existing
no-metrics fixtures as the backward-compat case), and the **mock producer** emitting the
metrics for every vehicle each tick in the running phases. Compliance harness stays green;
a payload with **none** of the new fields still validates (asserted directly in the harness
self-test).

## The consumer widget (richer tower, follow-on to #28)

Extend the existing class-aware standings tower
(`app/src/routes/tower/StandingsTower.svelte`) to surface the new metrics, config-gated so
a broadcaster chooses the tower's density:

- **Interval column** — `interval_ahead` rendered as a second timing cell beside the
  existing gap-to-leader cell (`+1.234`, `—` for the leader / undetermined). Rendered
  verbatim in both the inline and grouped layouts (producer-owned; not re-derived).
- **Pit indicator** — `pit_stops` count, and an **IN PIT** state on the row when `in_pit`
  is true (a `data-in-pit` hook + a compact badge/tint), so a car in the pits reads at a
  glance.
- **Tire indicator** — the `tire_compound` label in a compact chip, with an optional wear
  bar driven by `tire_wear` (clamped to `[0, 1]`).
- **Fuel indicator** — a compact `fuel` bar (clamped to `[0, 1]`).
- **Config-gated density.** A new per-widget **`towerMetrics`** knob —
  `{ interval, pit, tire, fuel }` booleans — mirrors the `driverInfo` pattern exactly:
  normalized onto every widget for a uniform shape, only the tower reads it. Default:
  **`interval` on** (the headline LMU upgrade, pairing with the existing gap column),
  **`pit` / `tire` / `fuel` off** (opt-in endurance density), mirroring how `driverInfo`
  ships a sensible subset on. Surfaced in `/config` as a "tower metrics shown" checkbox
  group (like the on-board HUD's "driver info shown").
- **Preserves everything the tower already does.** The class-aware inline/grouped layouts,
  the `?class=` filter, the leader's LEADER / pole-lap cell, the subject-keyed re-cut
  shine, and the `data-motion` gating are all untouched — the metrics render inside the
  shared `towerRow` snippet so both layouts get them identically.
- **Idle / absence tolerance.** Every metric cell renders only when its field is present
  and its toggle is on; a tower handed no metrics (older producer) looks exactly like
  today. Partial/garbage field types are tolerated without throwing — the same *dumb
  overlay* discipline as the rest of the tower.

Wiring mirrors the existing tower/onboard structure:
- `overlayConfig.js` gains `TOWER_METRICS_DEFAULTS` + `towerMetrics` normalization (like
  `ONBOARD_DEFAULTS` / `driverInfo`), and the `towerMetrics` field is added to every widget
  in `DEFAULT_CONFIG`.
- `StandingsTower.svelte` gains a `metrics` prop (the resolved `{ interval, pit, tire, fuel }`
  toggles) and renders the new cells in `towerRow`, plus small `resolveMetrics`-style module
  helpers for the defensive normalization.
- `AllView.svelte` passes `normalized.widgets.tower?.towerMetrics` into the tower.
- `TowerPage.svelte` parses a `?metrics=` comma list (`interval,pit,tire,fuel`) for the
  standalone route, mirroring the 0.6.0 `?unit=mph` knob and the existing `?class=`.
- `ConfigPage.svelte` adds the tower-only "tower metrics shown" control.

## Testing (per CONTRIBUTING — fixture-based behavioral)

- **Field presence matrix**: all six present / absent / partial (e.g. tire but no fuel)
  each render a sensible row; absent → the tower renders exactly as today, never a crash.
- **Interval column**: `interval_ahead` renders verbatim in inline and grouped layouts;
  leader / undetermined → `—`; tolerated when absent (older producer).
- **Config gating**: with `towerMetrics.interval` off the interval cell is absent from the
  DOM; with `pit`/`tire`/`fuel` on their indicators appear; default config shows the
  interval column and hides the strategy cluster.
- **Pit**: `pit_stops` count renders; `in_pit: true` marks the row (`data-in-pit`) and
  shows the IN PIT state; absent/false → on-track (no badge).
- **Tire / fuel bars**: fill widths track their `[0, 1]` values; values are clamped
  (out-of-range and garbage tolerated); absent → the indicator is hidden.
- **/all composition**: the tower reads the metrics from `snapshot.vehicles[]` via the
  `towerMetrics` config without disturbing the other widgets; toggling the config removes
  the cells from the DOM.
- **Compliance harness stays green** — the additions are additive, `schemaVersion`
  unchanged; assert a spec-v1 payload with **none** of the new fields still validates.

## Risks

- **Batch creep on #20** — the temptation to land more of #20 (track position, positions
  gained/lost, race events) at once. Mitigation: this plan scopes *only* the six
  interval/pit/in-pit/tire/fuel fields; the rest is explicitly deferred. Keep the PR to
  those fields.
- **Tower row overcrowding** — six new possible cells on a 380 px row could crowd the
  driver name. Mitigation: the strategy cluster (pit/tire/fuel) is **off by default** and
  opt-in per broadcaster; only the interval column ships on by default, and the name cell
  keeps its `flex: 1 1 auto` truncation so it yields gracefully. The default tower widget
  width can be widened in `DEFAULT_CONFIG` if the interval column proves tight.
- **Un-consumed fields** — adding spec fields no widget reads. Mitigation: the richer
  tower ships in the same release; the mock is the reference producer that keeps every
  field exercised each tick.
