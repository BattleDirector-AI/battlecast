# 0.8.0 release plan — Spec v1.x track position + live sector position overlay

**Theme:** land the **fourth slice** of the additive spec v1.x field batch (#20) — per-vehicle
**track position** — paired with the consumer that needs nothing but the field itself, the
**live sector position overlay** (#63). Same additive mechanics proven in 0.5.0 (`session`),
0.6.0 (`subject.telemetry`) and 0.7.0 (richer vehicle metrics). Alongside it, the tower
**cycling** feature that 0.7.0's clamp only made safe, plus two small carry-overs.

**Milestone:** [0.8.0](https://github.com/BattleDirector-AI/battlecast/milestone/8)
· **Epic:** #31 (widget parity) · **Spec slice:** #123 (slice 4 of #20)
**Built on:** 0.7.0 (richer tower #111, tower slot clamp #116/#118), and
`docs/decisions/0003-tower-overflow-pinning-and-cycling.md`.

## Why this, why now

The epic's own critical path names track position as a next slice, and it is the field with
**two** waiting consumers — the track map (#29) and live sector position (#63).

The map is the more eye-catching of the two and was the original pick for this release. It
was **deferred** (see #124) because it needs a second, non-spec dependency the project has no
answer for: a per-track outline (SVG/path). That is a per-circuit *asset pipeline* question,
a different class of problem from "read a field the producer sends", and it should not sit on
the critical path of a data slice.

\#63 consumes the identical field with **no asset dependency at all**, so the slice still
ships next to a real reader — the rule that has held for every slice so far. When #124
resolves, #29 becomes pure widget work against a field that is already spec'd, mock-emitted,
and proven in production by #63.

## The additive contract (slice 4 of #20)

```jsonc
// on each `vehicle` entry — both fields OPTIONAL
{
  "slot_id": "car-1",
  "lap_fraction": 0.412,          // number [0,1) | null — normalized position around the lap.
                                  // 0 = start/finish, increasing in the direction of travel.
  "position_xy": { "x": 1, "y": 2 } // optional; producer-defined coordinate space.
}
```

`lap_fraction` is normalized rather than metres so a consumer can place a car without knowing
the circuit length — the same discipline `tire_wear` / `fuel` / `throttle` take. Per the #29
research note it is **directly derivable in both sims** as `mLapDist / track_length`, so it is
cheap for a producer to supply.

Additive and backward-compatible: both optional, `additionalProperties` stays `true`, a payload
carrying neither still validates, so **`schemaVersion` stays `"1"`**.

**`lap_fraction` is primary; `position_xy` is optional.** The open question in #123 — whether
coordinates would serve better — only affects the *map*. #63 needs `lap_fraction` regardless,
because sector bucketing is a pure function of it and needs no geometry. So this release can
ship the field with confidence before #124 reports.

## The consumer widget (#63) — design decision

\#63's issue lists four *ideas*, not a design. They are materially different widgets, so this
release picks one.

**Decision: the sector-bucketed progress strip.** Cars grouped by the sector they are currently
in, ordered by progress within that sector, with the on-camera subject highlighted.

Three reasons:

1. **It is what the issue's own acceptance criteria already describe** — *"given a fixture with
   per-car lap fractions spanning sectors, the overlay buckets cars into the correct sector and
   orders them by progress; ties/edge-of-sector handled explicitly."* The other ideas have no
   acceptance criteria written.
2. **It is derivable from this slice alone.** Bucketing and ordering are pure functions of
   `lap_fraction`.
3. **It is the honest one.** "Who is fastest through S1 *right now*" is a **timing** claim, not
   a position claim. Spec v1 carries `sector_times` for **completed** sectors only; there is no
   elapsed-time-in-current-sector field. Deriving one overlay-side would mean the overlay timing
   the race — precisely what *dumb overlay, smart producer* forbids (ADR 0002). Live purple/green
   sector indicators are the same problem. Both are a later slice, if the producer can supply the
   timing.

So 0.8.0 ships the **live positional** picture — genuinely impossible today, since `sector_times`
only describes finished laps — and leaves the live *timing* picture to a slice that carries the
data for it.

### Sectors are an optional enhancement, not a requirement

**The strip renders whatever sector information the producer provides, and nothing more.** This
follows directly from the existing contract rather than from any sim's capabilities: every field
in this slice is optional, `sector_times` is already documented as *"may be empty if no lap has
been completed yet"*, and SPEC's versioning policy requires consumers to tolerate absent fields.
A widget that *requires* an optional field is simply not honoring the contract.

Two render modes, one widget:

- **Ungrouped (baseline)** — every car ordered by `lap_fraction`, subject highlighted. Needs
  nothing but this slice's field.
- **Grouped** — the same ordering, partitioned into sector bands, when sector information is
  present.

Same discipline every other widget already follows: the on-board HUD tolerates absent
`telemetry`, Race Control tolerates absent `session`, the tower tolerates absent metrics. This is
not a special case, and the ungrouped path is the **baseline tested first-class**, not a fallback.

**Where sector data comes from is the producer's problem.** If a producer's data source exposes
sector times, it emits them; if it does not, the producer either derives them (it holds
authoritative race state and can watch lap-fraction crossings itself) or omits them and the
overlay renders ungrouped. battlecast does not model any particular sim's limitations, and this
plan should not be read as designed around one — *dumb overlay, smart producer* means the spec is
the contract and conformance is the producer's side of it.

**Sector count comes from the data, never a constant.** `sector_times` is documented as *"Length
matches the track's sector count"*, so read the count off any vehicle with a completed lap. Three
is not a safe assumption: sector count is a property of the circuit, varies by track, and four is
common. When no vehicle has completed a lap the count is simply unknown — render ungrouped until
it is known.

Sector *boundaries* are a further gap: the spec carries no boundary positions, so mapping
`lap_fraction` to a sector needs either a producer-supplied boundary list (a candidate additive
field, deferred until a consumer proves it necessary) or a presentation default of even
divisions. Default to even divisions and document it plainly as an overlay assumption — this is
one of the few places the overlay decides something, so keep the rule obvious and overridable
rather than clever.

Note this makes even-division bucketing an *approximation*: real sector boundaries are rarely
evenly spaced. Grouped mode is therefore honest about ordering (which is exact, from
`lap_fraction`) but approximate about which band a car sits in near a boundary. If that proves
misleading in practice, the fix is a producer-supplied boundary field, not cleverer maths.

Edge-of-sector ties are explicit in the acceptance criteria and need a defined rule: order by
`lap_fraction` descending, `slot_id` as the stable tie-break (never driver name — two drivers
can share one, per ADR 0002).

## Carry-overs

### #116 — tower cycling

Implement ADR 0003 as accepted: `maxRows: 'auto'`, `cycle: { enabled, perPageSeconds, pinTop,
pinScope, pinSubject }`. Measured row budget, page membership frozen per turn, no page indicator,
no pause under caution, 4s floor, `data-motion` gating. Grouped mode stays clamp-only.

Two items the ADR left open, to settle here: whether `hideWhenIdle` interacts with cycling, and
whether the window cursor resets on a session-phase change.

### #115 — config staleness

Two halves, deliberately different sizes:

- **Cache header (do first, independently).** `sendJson` (`server/lib/respond.js`) sets no
  `Cache-Control`, unlike the logo assets which set `no-cache` (`createApp.js:147`). One line;
  makes a manual Browser Source refresh reliably work. It is also a **prerequisite** for the
  reload below — polling a heuristically-cached endpoint would poll a stale copy.
- **Live reload.** `AllPage.svelte:30` resolves config once in `onMount` and never re-reads.
  **Proposal: poll the profile endpoint on a modest interval and apply on change** — no new
  server infrastructure, and it degrades safely (a missed poll just delays a change). Push
  invalidation would mean a second live channel alongside the SSE feed for a low-frequency
  event, which is not worth the infrastructure.
  Apply a detected change **immediately and without transition**: a config edit is an operator
  action, not a broadcast reveal, and animating a geometry change mid-show would read as a glitch.

### #117 — widget opacity

**Per-widget plate alpha driving `--bc-plate`**, defaulted to today's `0.82` so existing profiles
render identically. Deliberately *not* element `opacity`, which dims text and borders and costs
legibility — a broadcaster wants a more transparent *plate*, not a faded widget. Note the
`backdrop-filter` blur is tuned against 0.82 and may read oddly at low alpha; keep blur constant
initially and revisit if it looks wrong.

## Sequencing

1. **#115 cache header** — independent, one line, unblocks reliable verification of everything else.
2. **#123 spec slice** — schema + SPEC + fixtures + mock, compliance green.
3. **#63 widget** — against the fixtures from (2).
4. **#116 cycling** — independent of the slice; can run in parallel with (2)/(3).
5. **#115 live reload** and **#117 opacity** — independent; land as they are ready.

(2) → (3) is the only hard ordering. Shape the fixtures in (2) around what (3) renders, so the
field does not ship subtly wrong.

## Testing (per CONTRIBUTING — fixture-based behavioral)

- Compliance fixtures for `lap_fraction` / `position_xy`: present, partial, and the absent
  backward-compat case asserted in the harness.
- #63: ordering from known lap fractions **with no sector data** (the baseline mode); bucketing
  and ordering when sector count is known, including a **four-sector** fixture; explicit
  edge-of-sector and tie cases; absent/partial/garbage positions render no empty plate.
- #116: row-budget derivation, pin de-duplication by `slot_id`, page-membership stability across
  ticks, field-shrink mid-cycle, cycling inert when the field fits.
- #115: cache header asserted on an API JSON response; reload applies a changed profile.
- Note `happy-dom` does no layout — anything geometric (row budget, clamp interaction) needs a
  real browser, as the #118 clamp did.

## Exit criteria (release gate)

- Compliance harness green; `schemaVersion` still `"1"`; a no-position payload still validates.
- #63 renders from the mock producer end-to-end with no per-track asset of any kind.
- #63 renders correctly from a payload carrying `lap_fraction` but **no `sector_times` at all** —
  ungrouped ordering, no empty plate, no console noise. A first-class case, not a fallback.
- #63 reads sector count from `sector_times` length rather than assuming three; a **four-sector**
  fixture is asserted.
- Cycling honors the configured `h` with a field larger than the row budget, verified in a real
  browser, with the clamp still bounding it.
- A config change applies to a running Browser Source without a manual refresh.
- `make dev-live` still drives everything against a real producer.

## Risks

- **#123's primary-field question resolves the wrong way.** If #124 concludes coordinates must be
  primary, `position_xy` is promoted later. Contained: both fields are optional and additive, and
  #63 does not care.
- **Reintroducing a hard dependency on optional data.** The contract makes sector information
  optional, so the implementation risk is drift: treating grouped as the "real" mode and
  ungrouped as a degraded afterthought. Guard by testing the ungrouped path first-class and
  gating the release on a no-`sector_times` payload rendering correctly.
- **Even-division bucketing is an approximation.** Real sector boundaries are rarely evenly
  spaced, so grouped mode can place a car in the wrong band near a boundary while its *ordering*
  stays exact. Documented as an overlay assumption; the fix if it misleads is a producer-supplied
  boundary field, not cleverer maths.
- **Cycling's page stability is the subtle part** (ADR 0003). Recomputing membership per tick
  makes rows jitter; the freeze rule needs a real test, not an eyeball.
- **Scope creep from #29.** If #124 resolves early and mid-release, the temptation will be to pull
  the map in. It should still wait — the asset pipeline is its own body of work.
